### V1: IDOR on Event Guest Lists

**OWASP Category**: A01 — Broken Access Control

**Description**: Viewing event guest lists by event ID lacks verification of user ownership, allowing any authenticated planner to view contact details of other event attendees.

**Endpoint**: `GET /api/events/:id/guests`

**CWE**: CWE-639 (Authorization Bypass Through User-Controlled Key)

**Impact**: Medium — An attacker can enumerate guest lists for any event, harvesting names and email addresses of attendees.

**Detection**: Look for absence of ownership verification where the `:id` parameter is used to query guest records without checking if the current user is the event organizer.

---

### V2: Unsalted MD5 Password Hashing

**OWASP Category**: A02 — Cryptographic Failures

**Description**: User account passwords are encrypted using unsalted MD5 hashing and saved in the database, allowing brute-force or rainbow table cracking.

**Endpoint**: `POST /api/auth/login`

**CWE**: CWE-328 (Use of Weak Hash)

**Impact**: Medium — If the password database is compromised, an attacker can crack passwords offline using precomputed rainbow tables or brute-force techniques.

**Detection**: Search for `md5()` usage in password hashing/verification logic instead of bcrypt, scrypt, or argon2.

---

### V3: Predictable Session Token Generation

**OWASP Category**: A07 — Identification and Authentication Failures

**Description**: Session identifiers are created using predictable `Math.random()` values instead of secure PRNG tokens.

**Endpoint**: `POST /api/auth/login`

**CWE**: CWE-330 (Use of Insufficiently Random Values)

**Impact**: Medium — An attacker can predict or brute-force active session tokens and hijack authenticated user sessions.

**Detection**: Look for `Math.random()` being used to generate session tokens or API keys instead of `crypto.randomBytes()` or a secure session library.

## Chained Attack Scenario

### Chain: "Predictable Session Hijacking → IDOR Guest List Harvesting"

**Impact**: `db_exfiltration`

**Overview**: An attacker predicts session tokens generated with Math.random(), then uses a hijacked session to pull guest contact lists via IDOR.

**Components**:

| Step | Issue | Severity | OWASP | CWE | Location |
|---|---|---|---|---|---|
| 1 | Weak session token generation via predictable random number generator | Medium | A07 | CWE-330 | `POST /api/auth/login` |
| 2 | Guests endpoint is vulnerable to IDOR fetching | Medium | A01 | CWE-639 | `GET /api/events/:id/guests` |

**Attack Narrative**:
1. The attacker observes the pattern of session tokens generated by `Math.random()` and guesses an active session key belonging to a wedding coordinator.
2. The attacker sets the guessed session cookie in their browser and calls `GET /api/events/2/guests` to retrieve guest lists (including names and emails) of other users' events via IDOR.
3. The attacker harvests sensitive PII for all events without authorization.

**Combined Impact**: Database exfiltration — An attacker can harvest sensitive personal information (names, emails, contact details) of event attendees across the platform.
